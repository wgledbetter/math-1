1 ## Adding A New Function With Known Gradients {#new_grad}
2 
3 If you have a function f and you know the gradients for it, it is straightforward to add the function to Stan's math library.
4 
5 
6 #### Templated function
7 
8 If your function is templated separately on all of its arguments and bottoms out only in functions built into the C++ standard library or into Stan, there's nothing to do other than add it to the Stan namespace. For example, if we want to code in a simple z-score calculation, we can do it very easily as follows, because the `mean` and `sd` function are built into Stan's math library:
9 
10 
11 ```cpp
12 #include <stan/math/rev/core.hpp>
13 
14 namespace stan {
15  namespace math {
16  vector<T> z(const vector<T>& x) {
17  T mean = mean(x);
18  T sd = sd(x);
19  vector<T> result;
20  for (size_t i = 0; i < x.size(); ++i)
21  result[i] = (x[i] - mean) / sd;
22  return result;
23  }
24  }
25 }
26 ```
27 
28 This will even work for higher-order autodiff if you include the top-level header `<stan/math.hpp>`.
29 
30 #### Simple univariate example with known derivatives
31 
32 Suppose have a code to calculate a univariate function and its derivative:
33 
34 ```cpp
35 namespace bar {
36  double foo(double x);
37  double d_foo(double x);
38 }
39 ```
40 
41 We can implement foo for Stan as follows, making sure to put it in the `stan::math` namespace so that argument-dependnet lookup (ADL) can work. We'll assume the two relevant functions are defined in `my_foo.hpp`
42 
43 ```cpp
44 #include "my_foo.hpp"
45 #include <stan/math/rev/core.hpp>
46 
47 namespace stan {
48  namespace math {
49  double foo(double x) {
50  return bar::foo(x);
51  }
52 
53  var foo(const var& x) {
54  double a = x.val();
55  double fa = bar::foo(x_d);
56  double dfa_da = bar::d_foo(a);
57  return precomp_v_vari(fa, x.vi_, dfa_da);
58  }
59  }
60 }
61 ```
62 
63 There are similar functions `precomp_vv_vari` for two-argument functions and so on.
64

67 #### Functions of more than one argument
68 
69 For most efficiency, each argument should independently allow `var` or `double` arguments (`int` arguments don't need gradients---those can just get carried along). So if you're writing a two-argument function of scalars, you want to do this:
70 
71 ```cpp
72 namespace bar {
73  double foo(double x, double y);
74  double dfoo_dx(double x, double y);
75  double dfoo_dy(double x, double y);
76 }
77 ```
78 
79 This can be coded as follows
80 
81 ```cpp
82 #include "my_foo.hpp"
83 #include <stan/math/rev/core.hpp>
84 
85 namespace stan {
86  namespace math {
87  double foo(double x, double y) {
88  return bar::foo(x, y);
89  }
90 
91  var foo(const var& x, double y) {
92  double a = x.val();
93  double fay = bar::foo(a, y);
94  double dfay_da = bar::foo_dx(a, y);
95  return return precomp_v_vari(fay, x.vi_, dfay_da);
96  }
97 
98  var foo(double x, const var& y) {
99  double b = y.val();
100  double fxb = bar::foo(x, b);
101  double dfxb_db = bar::foo_dy(x, b);
102  return return precomp_v_vari(fxb, y.vi_, dfxb_db);
103  }
104 
105  var foo(const var& x, const var& y) {
106  double a = x.val();
107  double b = y.val();
108  double fab = bar::foo(a, b);
109  double dfab_da = bar::foo_dx(a, b);
110  double dfab_db = bar::foo_dy(a, b);
111  return precomp_vv_vari(fab, x.vi_, y.vi_, dfab_da, dfab_db);
112  }
113  }
114 }
115 ```
116 
117 Same thing works for three scalars.

120 #### Vector function with one output
121 
122 Now suppose the signatures you have give you a function and a gradient using standard vectors (you can also use Eigen vectors or matrices in a similar way):
123 
124 ```cpp
125 namespace bar {
126  double foo(const vector<double>& x);
127  vector<double> grad_foo(const vector<double>& x);
128 }
129 ```
130 
131 Of course, it's often more efficient to compute these at the same time---if you can do that, replace the lines for `fa` and `grad_fa` witha more efficient call. Otherwise, the code follows the previous version:
132 
133 ```cpp
134 #include "my_foo.hpp"
135 #include <stan/math/rev/core.hpp>
136 
137 namespace stan {
138  namespace math {
139  vector<double> foo(const vector<double>& x) {
140  return bar::foo(x);
141  }
142 
143  vector<var> foo(const vector<var>& x) {
144  vector<double> a = value_of(x);
145  double fa = bar::foo(a);
146  vector<double> grad_fa = bar::grad_foo(a);
147  return precomputed_gradients(fa, x, grad_fa);
148  }
149  }
150 }
151 ```
152 
153 The `precomputed_gradients` class can be used to deal with any form of inputs and outputs. All you need to do is pack all the `var` arguments into one vector and their matching gradients into another.
154 
155 #### Functions with more than one output
156 
157 If you have a function with more than one output, you'll need the full Jacobian matrix. You create each output `var` the same way you create a univariate output, using `precomputed_gradients`. Then you just put them all together. This time, I'll use Eigen matrices assuming that's how you get your Jacobian:
158 
159 ```cpp
160 namespace bar {
161  Eigen::VectorXd foo(const Eigen::VectorXd& x);
162  Eigen::MatrixXd foo_jacobian(const Eigen::VectorXd& x);
163 }
164 ```
165 
166 You can code this up the same way as before (with the same includes):
167 
168 ```cpp
169 namespace stan {
170  namespace math {
171  Eigen::VectorXd foo(const Eigen::VectorXd& x) {
172  return bar::foo(x);
173  }
174  Eigen::Matrix<var, -1, 1> foo(const Eigen::Matrix<var,-1,1>& x,
175  std::ostream* pstream__) {
176  // extract parameter values from x
177  const var * x_data = x.data();
178  int x_size = x.size();
179  vector<var> x_std_var(x_data, x_data + x_size );
180  VectorXd a = value_of( x );
181 
182  // evaluate f
183  VectorXd f_val = bar::foo(a);
184  // f_val_jacobian[i][j] is the partial derivative of f_i w.r.t. parameter j
185  vector<vector<double>> f_val_jacobian = bar::foo_jacobian(a);
186 
187  int f_size = f_val_jacobian.size();
188  Eigen::Matrix<var, -1, 1> f_var_jacobian(f_size);
189  for (int i=0; i < f_size; i++) {
190  f_var_jacobian(i) = precomputed_gradients(f_val(i), x_std_var, f_val_jacobian[i]);
191  }
192  return f_var_jacobian;
193  }
194  }
195 }
196 ```
197 
198 We could obviously use a form of `precomputed_gradients` that takes in an entire Jacobian to remove the error-prone and non-memory-local loops (by default, matrices are stored column-major in Eigen).
